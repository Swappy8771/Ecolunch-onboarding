# 04 — Validation Center

---

## Purpose

The Validation Center is the global admin review queue. Every piece of data or document that a caterer submits — or that gets processed by Smart Import — creates a validation item here, waiting for an admin to review it.

Think of it as the admin's inbox for everything that needs a decision.

---

## Who Uses This Section

| Role | What They Do Here |
|------|-------------------|
| **Admin** | Reviews every submission. Approves, rejects, or requests corrections. This is where admin spends most of their active time. |
| **Caterer** | ❌ No access to this queue. They only see the outcome — approved, rejected, or a correction request sent via EcoLoop. |

---

## What Creates a Validation Item

A validation item is created automatically whenever:

| Trigger | Type Created |
|---------|-------------|
| Caterer edits profile fields and submits | `profile` |
| Caterer submits bank details | `banking` |
| A document is uploaded to the vault | `document` |
| Smart Import engine finishes extracting fields | `smart_import` |
| A contract is signed via Dropbox Sign | `contract` |
| A menu is submitted by caterer | `menu` |
| A school/daycare/camp establishment is added | `establishment` |
| Admin configures a module or pricing | `module` / `pricing` |
| A go-live checklist item fails | `golive` |

---

## Validation Item — Full Detail View

When admin opens a validation item, they see:

| Field | Description |
|-------|-------------|
| Type | What kind of submission this is |
| Priority | Critical / High / Medium / Low |
| Caterer | Which caterer this belongs to |
| Section | Which onboarding section this comes from |
| Data Snapshot | The exact values submitted, captured at review time |
| Linked Document | If a file is involved, a link to it in the vault |
| Smart Import Job | If this was generated by Smart Import, a link to the job |
| Internal Notes | Admin-only notes (caterer never sees these) |
| Audit History | All previous actions on this item |

### For Smart Import Items — Extra Fields

| Field | Description |
|-------|-------------|
| Source Document | Which document the data was extracted from |
| Target Section | Which EcoLunch section the field maps to |
| Detected Field Name | What the extractor identified |
| Current Value | What is currently in the system for this field |
| Proposed Value | What Smart Import extracted |
| Confidence Score | 0.0 to 1.0 — how certain the extraction was |
| Field Status | Confirmed / Edited / Rejected |

```
Pending Review
     │
     ▼
  In Review  ──────────────────────────────────┐
     │                                         │
     ├──► Approved                             │
     │                                         │
     ├──► Rejected                             │
     │                                         │
     └──► Correction Requested ──► (caterer fixes) ──► new Pending Review item
                                                          │
                                                   admin reviews again
                                                          │
                                              Approved or Rejected
```

---

## Priority Levels

| Priority | When Assigned | Examples |
|----------|--------------|---------|
| Critical | Cannot go live without resolution | Banking info, signed contracts, go-live blockers |
| High | Blocks a significant section | Document approvals, Smart Import fields, module config |
| Medium | Important but non-blocking for other stages | Profile changes, menu submissions, establishment adds |
| Low | Minor or informational | Internal document updates |

---

## Admin Actions Per Item

| Action | What Happens |
|--------|-------------|
| **Approve** | Item status → `approved`. Related checklist item re-evaluated. Audit log created. |
| **Reject** | Item status → `rejected`. Admin must provide a reason. Audit log created. |
| **Request Correction** | Item status → `correction_requested`. A `corrections` record is created. An EcoLoop ticket is automatically created and caterer is notified. |
| **Add Internal Note** | A note is saved, visible only to admin. Caterer never sees it. |
| **Send via EcoLoop** | Opens a new EcoLoop ticket or links to an existing one for this caterer. |
| **View History** | Shows the full audit trail for this specific item. |

---

## What Happens After Approval

Approving a validation item is not just a status change — it triggers downstream evaluations:

1. The relevant `caterer_onboarding_files` section status is updated
2. The relevant `golive_checklist_items` are re-evaluated
3. If all go-live items are now `complete`, the Go-live Monitor shows the caterer as "Ready"
4. An `audit_log` entry is created for the approval

---

## Correction Request Flow — Step by Step

1. Admin reviews item → clicks "Request Correction"
2. Admin fills in: what needs to be fixed, which section, priority level
3. System creates a `corrections` record linked to the validation item
4. System auto-creates an EcoLoop ticket linked to the correction
5. Caterer receives notification in their portal (and optionally via email)
6. Caterer opens the EcoLoop ticket, reads the correction request
7. Caterer fixes the issue in the relevant section and resubmits
8. A new validation item is created (type: same as original)
9. Admin reviews the new item
10. If approved: the `corrections` record status → `resolved`, EcoLoop ticket → `closed`

---

## Admin vs Caterer Role Summary

| Step | Admin | Caterer |
|------|-------|---------|
| Submitting data | ❌ | ✅ Submits via their portal |
| Reviewing submitted data | ✅ Approves / Rejects / Requests correction | ❌ |
| Seeing validation queue | ✅ Full queue across all caterers | ❌ |
| Seeing correction request | Creates it | Receives it via EcoLoop |
| Fixing and resubmitting | ❌ | ✅ Fixes in their portal section |
| Final approval decision | ✅ Only admin can approve | ❌ |
