The bridge — where the two databases meet

Transactional outbox / saga

Banking is in PostgreSQL, but its review status and approval queue live in MongoDB. So submitting banking touches both databases. We never split a write across two databases directly — we use an outbox the system drains reliably.

caterer_banking → caterer ·cross-DB caterer_banking → documents ×3 ·cross-DB validation_item → caterer_banking ·cross-DB
STEP 1

Save the banking row and an outbox note in one PostgreSQL transaction — both land, or neither.

STEP 2

A background relay worker reads unprocessed outbox notes.

STEP 3

It creates the validation_item + sets banking_status in MongoDB, idempotently.

STEP 4

The note is marked done. A crash simply retries — nothing lost or duplicated.

Rules that never bend

›Tenant isolation. Every record carries caterer_id; a caterer sees only its own data.
›Banking is never returned raw. Only the masked last-4 leaves the API.
›Smart Import never auto-applies. A human confirms every field.
›Internal notes stay internal. The caterer never sees them in chat.
›Go-live is all-or-nothing and only an admin can approve it.
